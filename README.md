# Проект 

Проект **AuthApplication** представляет собой backend-приложение – собственную систему аутентификации и авторизации.

# Схема структуры управления ограничениями доступа

Система управления ограничениями доступа в приложении реализована с применением следующих сущностей:

**Пользователь (User)** — зарегистрированный пользователь.
**Группа (Group)** — группа пользователей с набором разрешений ("Администратор", "Модератор", "Учитель", "Ученик").
**Разрешение (Permission)** — конкретное право, которое определяет, какие действия может выполнять пользователь 
(например, "удалять пользователя", "создавать домашнее задание" и пр.).

- правила доступа:

1. Пользователь.

Базовый уровень доступа, позволяющий пользователю просматривать свой профиль.
Пользователь имеет доступ к данным других пользователей лишь в усеченном формате. Таким образом реализован 
ограниченный доступ к персональным данным.

2. Модератор.

Имеет доступ к просмотру (в сокращенном виде) информации о других пользователях и добавлению новых пользователей.
Не может обновлять профили других пользователей и удалять их аккаунты.

3. Администратор.

Полный доступ ко всем действиям. Может регистрировать, просматривать, редактировать и удалять профили других
пользователей, а также управлять домашними заданиями.

4. Учитель.

Имеет полный доступ к домашним заданиям и правам управления ими (создавать, редактировать, удалять).
Учителя могут создавать и редактировать учебные материалы, но не имеют доступа к общим пользовательским данным.

5. Ученик.

Имеет ограниченный доступ к ресурсу (может просматривать домашнее задание).


- взаимосвязь компонентов:

1. Пользователь и роль.
Пользователь может принадлежать к одной или нескольким группам, определяющим его права доступа.

2. Роль и разрешение.
Каждой роли назначается набор разрешений, которые определяют, какие действия возможны для пользователей этой роли.

- реализация:

Для разграничения доступа используются кастомные классы разрешений (IsAdminOrModerator, IsAdminOrProfileOwner, 
IsAdminOrModeratorOrProfileOwner), которые проверяют принадлежность пользователя к определенной роли и наличие 
конкретных разрешений.
Доступ к представлениям контролируется с помощью аттрибута **permission_classes**, в котором устанавливаются необходимые
разрешения.

В приложении предусмотрено также создание вымышленных объектов бизнес-приложения для демонстрации работы данной системы.
Такая концепция реализована при помощи mock-view, при обращении к которым выдаются потенциальные объекты или ошибки.

В частности, имитируется работа с объектом "домашнее задание" на основе разграничения прав доступа для групп 
пользователей: "Учитель" и "Ученик".

Учителя могут создавать, редактировать и удалять домашние задания, ученики могут лишь их просматривать для дальнейшего
решения и отправки.

- безопасность:

Все операции, связанные с изменениями данных, защищаются классами разрешений.
Неавторизованные пользователи получают ошибку 401 (Unauthorized), а пользователи без соответствующих прав — 
ошибку 403 (Forbidden).
